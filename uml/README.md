# UML-диаграммы ВКР

**Тема:** Разработка системы управления контентом в сфере маркетинга на базе Strapi и Astro

## Описание системы

Разрабатываемый продукт — это open-source шаблон маркетингового сайта, позволяющий компаниям быстро развернуть веб-присутствие с управляемым контентом. Система построена на двух компонентах: **Strapi** (headless CMS, бэкенд) и **Astro** (статический генератор сайтов, фронтенд).

Strapi предоставляет административную панель для управления контентом и REST API с автоматически генерируемой OpenAPI-документацией. Фронтенд на Astro использует режим статической генерации (SSG): при сборке он запрашивает данные из Strapi API и генерирует готовые HTML-страницы. Это обеспечивает максимальную производительность для посетителей — страницы не требуют обращения к серверу во время просмотра.

Система поддерживает три типа пользователей: **Администратор/Разработчик** (настройка системы, управление ролями, деплой), **Контент-менеджер** (создание и публикация контента), **Посетитель** (чтение сайта).

## Содержание диаграмм

### 01 — Варианты использования (Use Case)

Три актора: Администратор/Разработчик, Контент-менеджер, Посетитель.

- **Администратор** управляет ролями, плагинами, пользователями, запускает сборку сайта и генерацию API-клиента.
- **Контент-менеджер** создаёт и редактирует статьи (Article), проекты (Project), вакансии (Vacancy) и авторов (Author), загружает медиафайлы, публикует и снимает с публикации контент.
- **Посетитель** просматривает списки и детальные страницы статей, проектов и вакансий.

### 02 — Классов (Classes)

Диаграмма отражает среднюю детализацию:

- **Strapi-сущности** (`Article`, `Author`, `Project`, `Vacancy`, `Global`, `HomePage`) наследуют базовый класс `StrapiBaseEntity` (id, documentId, timestamps). Локализуемые типы дополнительно наследуют `LocalizedEntity`.
- **Связи:** `Article` и `Author` связаны отношением `manyToMany`. `Article`, `Project`, `Global` ссылаются на `UploadFile` (медиабиблиотека).
- **Фронтенд:** `StrapiApiClient` инкапсулирует HTTP-обращения к API. Страницы (`ArticlePage`, `ProjectPage`, `VacanciesPage`) зависят от клиента и отображают данные через компоненты-карточки.

### 03a — Последовательность: Публикация контента

Показывает полный цикл от аутентификации контент-менеджера до появления записи в базе данных со статусом Published:
логин → создание черновика → загрузка обложки → привязка автора → публикация (установка `publishedAt`).

### 03b — Последовательность: Чтение страницы

Показывает два этапа:

1. **Build-time:** `astro build` вызывает `getStaticPaths()`, запрашивает все опубликованные записи, для каждой получает детальные данные и генерирует HTML-файлы.
2. **Runtime:** Посетитель получает готовый HTML без обращений к Strapi API. Гидратируются только интерактивные UI-компоненты (`Header`, `Footer`, `HeroContent`).

### 04 — Состояний (State)

Жизненный цикл контентной единицы (применимо к Article, Project, Vacancy):

- **Draft** — черновик, недоступен через публичный API.
- **Published** — опубликован (`publishedAt` установлен), включается в сборку Astro.
- **Deleted** — удалён из базы данных.

Переходы: Draft ↔ Published (публикация/снятие с публикации), Draft/Published → Deleted.

### 05 — Деятельности: публикация контента (Activity)

Бизнес-процесс создания материала с swimlanes для четырёх участников: Контент-менеджер, Система (Strapi), CI/Разработчик, Система (Astro Build).

Ключевые точки принятия решений: валидация обязательных полей, готовность к публикации. Диаграмма отражает архитектурную особенность системы — новый контент появляется на сайте только после запуска `astro build`.

### 05b — Деятельности: форма заявки ⁽планируемый функционал⁾ (Activity)

Два параллельных бизнес-процесса:

1. **Посетитель** — заполняет контактную форму или форму отклика на вакансию. Включает клиентскую валидацию, отправку POST-запроса в Strapi API и отображение результата.
2. **Контент-менеджер** — просматривает входящие заявки в Admin UI, связывается с заявителями, отмечает заявки как обработанные.

Реализация требует добавления в Strapi двух новых content types: `ApplicationForm` и `VacancyApplication`, а также перевода фронтенда на SSR-режим или использования Astro API Routes.

## Просмотр диаграмм

### В VS Code / Cursor

1. Установить расширение **PlantUML** (`jebbs.plantuml`)
2. Открыть любой `.puml` файл
3. Нажать `Alt+D` для предпросмотра (или `Cmd+Shift+P` → "PlantUML: Preview Current Diagram")

Расширению требуется Java. Альтернативно — в настройках расширения указать `"plantuml.server": "https://www.plantuml.com/plantuml"` для серверного рендеринга без Java.

### Онлайн

Скопировать содержимое файла на [plantuml.com/plantuml](https://www.plantuml.com/plantuml/uml).
